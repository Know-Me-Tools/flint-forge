Flint Forge

The sovereign data & edge-compute plane of the Flint platform: a backend-agnostic REST + GraphQL database gateway, an in-database capability suite, and a polyglot WASM edge-function runtime — unified by one identity model and one governance boundary.

01

Executive Summary

Flint Forge is the data and edge-compute plane of the Flint platform. It sits behind flint-gate (ingress and authentication) and consumes flint-realtime-fabric (the change-data spine), exposing a uniform governance, identity, and realtime surface over a Postgres 18 backend and a WASM edge runtime. It replaces the Supabase pattern — PostgREST, pg_graphql, GoTrue, Realtime, Database Webhooks, and Edge Functions — with a sovereign, model-independent, Rust-native architecture in which orchestration and governance are the durable moat.

The platform is one Cargo workspace delivering three subsystems, named for the forge: raw material is quarried from the source, shaped on the anvil in place, and fired in the kiln into hardened tools.

The thesis

Three properties make this a moat rather than a clone: one identity injection powers both row-level security and trigger-side JWT forwarding; one durable capture feeds both webhooks and LLM jobs; one WASM host serves both edge functions and UAR skills. Every external effect — a webhook, an LLM call, an edge function's database access — is routed back through flint-gate and governed by Cedar, Keto, and Postgres RLS. Nothing escapes the boundary.

Architecture Principles

Flint Forge is built as a hexagonal (ports-and-adapters) clean architecture. The discipline is not decorative: it is what makes the backend swappable, the security boundary provable, and the build parallelizable across the KBD/PMPO loop. Three rules are load-bearing.

The dependency rule

Dependencies point inward only. The domain knows nothing of infrastructure; infrastructure knows nothing of interfaces. The compiler enforces this — adapter crates are simply absent from the [dependencies] of domain and application crates, so a violation does not compile.

Capability inversion at compile time

The crate dependency graph physically excludes write actuators and provider keys from the agent and component layers. Edge components receive only governed host capabilities; the in-DB LLM extension holds no provider keys; the Kiln data plane is compiled without a WASM compiler. These are structural guarantees, not runtime policies — the unsafe path does not exist in the binary.

The two convergence invariants

Where a naive design would grow surface, Flint Forge collapses it. Two mechanisms are each built once and consumed twice.

InvariantBuilt onceConsumed by
One in-transaction captureA trigger captures (change + origin JWT) from request.headers, plus a durable outboxflint_hooks (webhooks) & flint_llm (LLM jobs)
One WASM component hostWasmtime engine, ProxyPre cache, Cedar-gated linker, fuel/epoch limitsFlint Kiln (edge) & UAR Tier-2 skills
Why this matters for the build

Convergence is a cost-control decision, not just elegance. Each shared mechanism is one crate to test, one security review, one place where the JWT-capture or sandbox semantics can be wrong. The KBD plan (§09) sequences these shared crates first so both consumers inherit a proven foundation.

Cross-Cutting Foundations

Identity & the JWT injection

Postgres has no native knowledge of application identity; it is injected as transaction-scoped settings (GUCs). The single injection that powers row-level security also makes the JWT available to triggers — one mechanism, two consumers.

-- set per request transaction by fdb-postgres before any user statement
SET LOCAL ROLE authenticated;                          -- single pooled role (pooling-safe)
SET LOCAL "request.jwt.claims" = '{"sub":…,"role":…,"tenant_id":…}';   -- decoded → RLS
SET LOCAL "request.headers"    = '{"authorization":"Bearer <raw-jwt>"}'; -- raw token carrier
Outbound — Option-3 hybrid (ratified)

When the database calls outward, the call is authenticated with a service identity flint-gate trusts; the originating user JWT travels in X-Forge-Origin-JWT for identity/audit/Cedar, plus an HMAC body signature (X-Forge-Signature). Delivery never fails on user-JWT expiry, yet the origin identity stays attributable. A per-registration forward_jwt=true opts into raw-bearer forwarding for prompt-delivery cases.

Quality gates (CI-enforced, all crates)

No unwrap/expect in library crates (thiserror in libs, anyhow only at binary edges); clippy::pedantic + deny(warnings); #[non_exhaustive] on public enums; newtype IDs with #[repr(transparent)]; tracing spans across every port boundary; pinned MSRV; semver discipline on forge-domain; Dagger CI. Never log JWT payloads, claims, relation tuples, or tenant identifiers.

Flint Quarry — REST / GraphQL DB API Gateway

Axum 0.8.8async-graphql 7deadpool-postgrestokio-postgrestonicarc-swap

Quarry exposes a uniform REST and GraphQL surface over Postgres 18. Query and mutation GraphQL pass through to pg_graphql (mature, RLS-aware, in-session); subscriptions are served by an async-graphql sibling schema sourced from the realtime fabric. The GraphQlExecutor port is the reversibility seam: a future uniform multi-backend dialect is a second adapter, leaving callers untouched.

Ports (fdb-ports)

#[async_trait] pub trait SchemaProvider   { async fn introspect(&self) -> Result<Vec<TableMeta>, BackendError>;
                                            fn subscribe_ddl(&self) -> watch::Receiver<SchemaVersion>; }
#[async_trait] pub trait RestExecutor     { async fn execute(&self, q: RestQuery, rls: &RlsContext) -> Result<RestResult, BackendError>; }
#[async_trait] pub trait GraphQlExecutor  { async fn execute(&self, r: GraphQlRequest, rls: &RlsContext) -> Result<Value, BackendError>; }
#[async_trait] pub trait ChangeStreamSource{ async fn watch(&self, s: SubscriptionSpec, who: &RlsContext)
                                              -> Result<BoxStream<'static, Result<ChangeEvent, StreamError>>, StreamError>; }

GraphQL — hybrid by operation type

enum ChangeOp { INSERT UPDATE DELETE UPSERT }
type TChangePayload { op: ChangeOp!  record: T  old_record: T }
type Subscription { tChanges(filter: TFilter): TChangePayload! }
Why passthrough, not reimplement

pg_graphql already implements Relay connections, FK-nested relationships, the filter/order grammar, and cursor pagination — and resolves in-session so RLS applies identically to REST. Reimplementing in async-graphql buys a uniform multi-backend dialect we do not yet need, at multi-month cost plus a permanent parity tax. The port keeps that option reversible; we spend engineering only on the differentiated subscription path.

Subscriptions & RLS enforcement

The WAL bypasses RLS — a change event contains a row regardless of who may see it. Quarry therefore enforces visibility itself, in two stages, and this is the genuinely hard part of the system.

  1. Source. fdb-realtime is a gRPC client of the fabric's WatchEntityType(tenant, entity_type, filter) stream. It runs no CDC of its own.
  2. Subscribe-time gate (coarse). One Keto check(subject,"view",entity_type|tenant), cached for the subscription's lifetime.
  3. Per-event authoritative filter. For each change, re-query the changed row as the subscriber (claims set, RLS applies). Deliver only if it survives — the same technique Supabase Realtime uses, and the only path faithful to query-time RLS semantics.
  4. Opt-in predicate pushdown (per hot table). Translate the RLS predicate to a payload filter, skipping the re-query. Operator-accepted risk, off by default: predicate drift silently leaks or drops rows.

REST surface & pgvector

PostgREST-compatible: GET/POST/PATCH/DELETE /<schema>.<table> with select, filter operators (eq,neq,gt,gte,lt,lte,like,ilike,in,is,cs,fts), order, range headers, and Prefer semantics. POST /rpc/<fn> invokes Postgres functions. pgvector similarity ships first as /rpc/<fn> over a SQL function (ORDER BY embedding <-> $q LIMIT k); a GraphQL vector field is a later sibling-schema enhancement.

Flint Anvil — pgrx Extension Suite

pgrxPostgres 18pg_netpgcryptobackground workers

Four extensions, each a workspace crate built with cargo pgrx and shipped into the prebuilt Postgres 18 image. Anvil shapes data in place: it provides the auth vocabulary RLS policies are written in, dispatches webhooks with forwarded identity, binds LLM inference into the database, and holds every secret encrypted at rest.

flint_auth — JWT / RLS context helpers

Defines the auth schema so policy authors never touch current_setting directly. Ships no policies — it provides the vocabulary policies are written in.

CREATE FUNCTION auth.jwt()    RETURNS jsonb LANGUAGE sql STABLE AS
  $$ SELECT coalesce(current_setting('request.jwt.claims', true), '{}')::jsonb $$;
CREATE FUNCTION auth.uid()    RETURNS text  LANGUAGE sql STABLE AS $$ SELECT auth.jwt()->>'sub' $$;
CREATE FUNCTION auth.role()   RETURNS text  LANGUAGE sql STABLE AS $$ SELECT coalesce(auth.jwt()->>'role','anon') $$;
CREATE FUNCTION auth.bearer() RETURNS text  LANGUAGE sql STABLE AS          -- raw token for forwarding
  $$ SELECT current_setting('request.headers', true)::json->>'authorization' $$;

flint_hooks — webhook dispatch

A gateway-managed registry drives one generic SECURITY DEFINER trigger per hooked table. SECURITY DEFINER changes the privilege context (so the function can reach net.* and the registry) but current_setting still returns the session GUCs — so the trigger reads the originating JWT while running privileged.

CREATE TABLE flint.webhooks (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  schema_name text, table_name text, events text[],          -- {'INSERT','UPDATE','DELETE'}
  target_url text,                                            -- routes through flint-gate
  forward_jwt boolean DEFAULT false,                          -- Option-1 opt-in (else Option-3)
  custom_headers jsonb DEFAULT '{}', secret text,             -- HMAC key
  tier text DEFAULT 'standard',                               -- 'standard'(pg_net) | 'durable'(outbox)
  active boolean DEFAULT true, timeout_ms int DEFAULT 5000);

CREATE TABLE flint.webhook_outbox (                           -- durable tier
  id bigserial PRIMARY KEY, webhook_id uuid, payload jsonb, headers jsonb,
  status text DEFAULT 'pending', visible_at timestamptz DEFAULT now(), retry_count int DEFAULT 0);

Flint Ember — flint_llm (in-database LLM via liter-llm)

liter-llmpgrx BGWroutes through flint-gate / UAR

Inference is bound into Postgres but routes inward to flint-gate/UAR, never directly to providers. The database holds a flint-gate credential, never provider keys; every in-DB call inherits Cedar policy, rate-limiting, cost attribution, and model routing, and may be served by candle-vllm/RunPod. Two surfaces with a hard boundary between them.

The hazard the design avoids

A synchronous LLM call inside a write trigger holds the row locks for the entire model latency. A bulk insert through a row-level sync trigger becomes N serial LLM calls in one transaction → backend/connection-pool exhaustion → whole-database stall. No mature project does this (pgai, pgEdge, pg_vectorize all use a worker + queue). The async surface is the default for triggers; the sync surface is gated and explicit.

Surface 1 — synchronous (read / explicit path only)

llm.embed(input text, model text DEFAULT 'default') RETURNS vector
llm.complete(prompt text, opts jsonb DEFAULT '{}')   RETURNS text

liter-llm runs on a dedicated runtime thread touching no Postgres internals; the backend thread blocks under a hard timeout + periodic CHECK_FOR_INTERRUPTS()/WaitLatch so statement_timeout and pg_cancel_backend work. EXECUTE granted narrowly. Legitimate only when the output gates the write or is required-at-commit — rare, low-volume, operator-opt-in per table. Never the default in a write trigger.

Surface 2 — asynchronous enrichment (default for triggers)

CREATE TABLE llm.jobs (
  id bigserial PRIMARY KEY, kind text,          -- 'embed' | 'summarize' | 'classify'
  schema_name text, table_name text, pk jsonb,
  source jsonb, target_column text, model text,
  origin_jwt text,                              -- captured at enqueue: attribution + Cedar
  status text DEFAULT 'pending', visible_at timestamptz DEFAULT now(), retry_count int DEFAULT 0);
-- dequeue: SELECT … FOR UPDATE SKIP LOCKED  (PGMQ pattern)

-- declarative (provisions column + enqueue trigger + HNSW index + worker config, index-like):
llm.enable_embedding(table regclass, column text, model text, dim int)
llm.enable_summary(table regclass, source_col text, target_col text, prompt_template text)

v1 scope: embeddings via Surface 2 first (safe, pairs with pgvector); summaries/completions follow once the worker + governor are proven; Surface 1 added when a real gating use case appears.

flint_vault — Sovereign Secret Store

pgrxXChaCha20-Poly1305KMS envelopeHKDF-SHA256

A pgrx extension that stores secrets of any kind — database passwords, external-service API keys (LLM providers, Stripe, SendGrid, …), connection strings, OAuth/refresh tokens, certificates, and arbitrary secret parameters — encrypted at rest in an ordinary Postgres table, so backups/WAL/replicas carry ciphertext only. It is the single sovereign store every Forge subsystem reads from; LLM keys (§4.3) are one consumer among many. It descends from the Supabase-Vault/pgsodium lineage but drops the deprecated pgsodium dependency and the file-only root key.

Crypto

XChaCha20-Poly1305 AEAD (24-byte nonce per row). The row id is bound in as associated data, so a ciphertext copied to another row fails authentication. The per-row working key is HKDF-SHA256(DEK, info = category ‖ key_id), so a category can be rotated independently.

Envelope encryption — the key improvement over the reference

The Data Encryption Key (DEK) is never a raw file. At postmaster start it is unwrapped from a Key Encryption Key (KEK) that lives in an external KMS — Azure Key Vault via managed identity (or AWS KMS / GCP KMS / Vault Transit) — through a KMS-agnostic unwrap command (the EDB-TDE PGDATAKEYUNWRAPCMD pattern: wrapped DEK on stdin → 32 raw bytes on stdout). The KEK never enters the DB or the process. Stakes: revoking the KEK renders every secret cryptographically dead (KMS kill-switch); rotating the KEK only rewraps the DEK — zero data re-encryption. A raw-DEK env var remains as the dev fallback. The DEK is held in secrecy::Secret, zeroized on drop, and is never selectable from SQL. _PG_init fails fast and loud if a configured key cannot be loaded.

Schema (in schema vault)

CREATE TYPE vault.secret_category AS ENUM
  ('api_key','password','connection_string','token','certificate','secret_param');

  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  category vault.secret_category NOT NULL,
  name text NOT NULL, description text NOT NULL DEFAULT '',

  scope text,                    -- optional tenant/environment scope
  secret text NOT NULL,          -- base64 ciphertext
  key_id uuid NOT NULL DEFAULT '…0001', nonce bytea NOT NULL,
  created_at timestamptz DEFAULT now(), updated_at timestamptz DEFAULT now(),

Access model — "available but not exposed" (two tiers)

Grounded in the wasmCloud secrets model and host-boundary credential injection (IronClaw):

  • In-process (DB consumers). flint_llm, flint_hooks, and flint-gate's service role resolve plaintext in the trusted Postgres process via SECURITY DEFINER functions vault.get_secret(name[,scope]) and vault.resolve_api_key(provider[,scope]), granted only to a flint_secret_reader role (flint_llm_worker inherits it) — never to PUBLIC, every call audited.
  • Edge (WASM, brokered — default for high-value secrets). A component never receives the raw value: it calls flint:llm or a host-mediated outbound, and Flint Kiln injects the secret at the host boundary (the key never enters WASM linear memory).
  • Edge (delegated — escape hatch, default-deny). When a component must read a value, flint:secrets.get(name) returns an opaque secret resource and reveal(secret) is a separate, Cedar-gated, audited call (§5.4).

Write path

vault.create_secret / update_secret are SECURITY DEFINER, revoked from PUBLIC, granted to a vault_admin role, and meant to be driven by the trusted host (flint-gate/Tauri/Axum) with statement logging disabled on that path (plaintext is a function argument — the one residual hole inherited from the reference).

Reconciliation — supersedes a prior decision

The earlier "provider keys live in flint-gate, never in the DB" is refined to: no secret is in the DB in plaintext. Secrets are ciphertext in flint_vault, the DEK is KMS-wrapped, and decryption happens only in the trusted DB process or trusted host — never to SQL clients or WASM. v1 resolution uses the SECURITY DEFINER SQL path (residual risk: a role that can assume flint_secret_reader can read plaintext, and it transits an SPI result); v2 hardens to a true in-process Rust entrypoint with no SQL round-trip.

v1 scope: crate compiles + packages against pgrx 0.18.1 / PG18 (flint_vault.so + generated SQL + control, verified). Open items — KEK provider, brokered-vs-delegated default, crypto lib (RustCrypto vs dryoc for mlock'd pages), wrapped-DEK location — are in §12.

Flint Kiln — WASM Edge Function Gateway

WasmtimeCraneliftwasi:http/proxyComponent ModelEd25519 / DIDCosign

Kiln compiles and runs signed, content-addressed WASM components — the deployable unit is a single .wasm, not a source directory. It mirrors Supabase Edge Functions but replaces Deno isolates with the standard WASM Component Model, giving stronger isolation, polyglot authoring, and AOT-to-native compilation. It shares its Wasmtime host with UAR's Tier-2 skills.

Runtime model

A bespoke Wasmtime host targets the standard wasi:http/proxy world — a component exports wasi:http/incoming-handler. Pre-instantiate once with ProxyPre; per request create a fresh Store and call ProxyPre::instantiate_async. Fresh linear memory per invocation means no cross-request state leakage; functions externalize state to flint:kv/DB. Epoch + fuel interruption enforce per-request timeouts; StoreLimits caps memory; the pooling allocator gives low-latency instances.

Compilation strategy — Cranelift, AOT, control/data-plane split

Wasmtime + Cranelift already emits native machine code; the dial is when and which backend. CompilationStrategy is a first-class config dimension.

StrategyUseMechanism
Cranelift-AOT (default, prod)registered functionsAt registration (post-verify) precompile_component → .cwasm; cache keyed (source_digest, target_arch, wasmtime_version); invocation deserializes native code, zero compile latency.
Winch (dev)fast iteratebaseline compile ~15–20× faster, code ~1.1–1.5× slower; x86_64. No Winch→Cranelift auto-tiering.
Pulley (fallback)no-codegen targetsportable interpreter; slowest; default where Cranelift has no backend.
.cwasm trust — an RCE surface

deserialize trusts the artifact; a tampered .cwasm runs as native code. Resolution: sign the source .wasm, AOT-compile only inside the trusted control plane, and treat the .cwasm as trusted because it was produced from a verified source in a controlled pipeline (optionally sealing it with a runtime key the data plane checks). A Wasmtime upgrade invalidates the cache → the pipeline recompiles on version bumps. Going "fully native" without the runtime would strip the sandbox, capability linker, and interruption — that is explicitly a non-goal.

Capability interfaces (WIT) — governed by Cedar

A component can do only what the host links into it. The Linker adds only the interfaces the function's signed manifest requests ∩ Cedar allows for that publisher (default-deny). A function never holds a raw DB connection or provider key — only governed host capabilities, so it cannot escape the boundary.

package flint:host@0.1.0;
interface db       { query: func(sql: string, params: list<json>) -> result<list<json>, error>; }
interface llm      { embed: func(input: string, model: option<string>) -> result<list<f32>, error>;
                     complete: func(prompt: string, opts: json) -> result<string, error>; }
interface kv       { get: func(k: string) -> option<list<u8>>; set: func(k: string, v: list<u8>); }
interface identity { origin-jwt: func() -> option<string>; claims: func() -> json; }
interface secrets  { get: func(name: string) -> result<string, error>; }   // Cedar-gated, per-function
world edge-function {
  export wasi:http/incoming-handler;
  import wasi:http/outgoing-handler;
  import db; import llm; import kv; import identity; import secrets;
}

Signing model & storage

Content-address every component (sha256 digest / IPFS CID). The registry maps name@version → digest; artifacts are immutable, name pointers mutable.

  • Sovereign default — Ed25519 detached signature over a manifest, keyed to the publisher's did:prometheus: identity, issued as a Kaia Verifiable Credential. The runtime verifies signature + DID + validity before instantiation and refuses unsigned or capability-over-requesting components.
  • OCI interop — Cosign/Sigstore: sign by digest, OIDC-keyless, signature + Rekor transparency entry alongside the artifact (SLSA alignment for components shared outside the sovereign boundary).
{ "publisher_did":"did:prometheus:…", "content_digest":"sha256:…",
  "capabilities":["flint:db","flint:llm","wasi:http/outgoing"],
  "version":"1.2.0", "not_before":"…", "not_after":"…" }     // granted = capabilities ∩ Cedar(publisher)

Storage is one ComponentStore port, content-addressed, with adapters: OCI registry (primary — wkg/cosign tooling, sign by digest never tag), IPFS (sovereign, CID-native), S3 (pragmatic), filesystem (dev).

The admin surface is the platform's highest-value target

Whoever can register a component can run code in the edge plane. Signing-key custody (Kaia / Key Vault) and per-publisher Cedar gating are non-negotiable, and the admin listener (control plane) must be locked down harder than the invocation path — ideally a separate listener with separate auth and policy.

Non-Goals

  • No end-user authentication (Kratos via flint-gate). No TLS termination (reverse proxy / flint-gate).
  • No second database backend built — the GraphQlExecutor/DatabaseBackend seams preserve the option.
  • No uniform cross-backend GraphQL dialect (deferred; reversible at the port).
  • No re-exposure of the fabric's own realtime client surface (the fabric owns it).
  • No synchronous LLM calls on the write hot path by default (Surface 2 async is the default; Surface 1 is gated/explicit).
  • No runtime-less "fully native" WASM execution — it would strip the sandbox, capability linker, and interruption; AOT keeps the runtime.
  • No secret of any kind stored in Postgres in plaintext, and no raw secret value exposed to edge components — secrets live encrypted in flint_vault (§4.4) under a KMS-wrapped DEK; inference still routes through flint-gate/UAR, and edge access is brokered or Cedar-gated reveal.
