Executive Overview

RFC-FORGE-A2UI-001

The Flint A2UI Component Registry is a domain-agnostic, metadata-driven system that enables AI agents to discover, select, compose, and render user interface components through structured JSON references.

Core Philosophy

Agents do not generate raw HTML/CSS. They reference constrained component identifiers with structured parameters, and the registry handles the rendering, data binding, and lifecycle management.

  • Security — No arbitrary markup injection
  • Consistency — All UIs follow the same design system
  • Composability — Components nest and combine predictably
  • Accessibility — Base primitives include ARIA roles
  • Performance — Pre-optimized rendering paths

Registry Schema

PostgreSQL + pgvector

The registry schema consists of six core tables stored in PostgreSQL, with pgvector embeddings for semantic component discovery.

Core Tables

TablePurposeKey Columns
componentsComponent definitionsid, name, category, json_schema, embedding, htmx_template
applicationsApp registrationsid, name, domain, config_json, embedding
design_systemsTheme tokensid, name, tokens_json, css_variables
bindingsDB field mappingsid, component_id, table_name, column_name, transform
eventsEvent definitionsid, name, payload_schema, handlers
embeddingsSemantic vectorsid, entity_type, entity_id, embedding vector(1536)

JSON Schema Example

{
  "type": "object",
  "properties": {
    "type": {"type": "string", "enum": ["button"]},
    "label": {"type": "string", "maxLength": 50},
    "variant": {"type": "string", "enum": ["primary","secondary","ghost"]},
    "size": {"type": "string", "enum": ["sm","md","lg"]},
    "action": {"type": "string"},
    "hx_post": {"type": "string", "format": "uri"},
    "hx_target": {"type": "string"},
    "hx_swap": {"type": "string", "enum": ["innerHTML","outerHTML","beforeend"]}
  },
  "required": ["type", "label"]
}

Database Bindings

Auto-binding

Components can be automatically bound to PostgreSQL tables and columns. When a table is registered, the registry suggests matching components based on column types and constraints.

Binding Configuration

{
  "table": "users",
  "column": "email",
  "component": "text_input",
  "transform": {
    "validation": "required|email",
    "placeholder": "user@example.com",
    "label": "Email Address"
  }
}

Auto-binding Trigger

CREATE TRIGGER auto_bind_component
AFTER INSERT ON information_schema.columns
FOR EACH ROW
EXECUTE FUNCTION suggest_component_binding();

Security Model

RLS + JWT

All registry tables use PostgreSQL Row Level Security (RLS) with JWT claims. The security model follows the principle of least privilege.

RLS Policies

CREATE POLICY component_read_policy ON components
FOR SELECT USING (jwt_claims ->> 'role' IN ('admin', 'developer', 'agent'));

CREATE POLICY component_write_policy ON components
FOR INSERT WITH CHECK (jwt_claims ->> 'role' = 'admin');
