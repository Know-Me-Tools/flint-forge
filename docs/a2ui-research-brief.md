## 1. Executive Summary

This research brief synthesizes six interrelated topics that form the conceptual and technical foundation for a **global A2UI (Agent-to-User Interface) component registry**. The central thesis is that agents will increasingly generate UI dynamically at runtime, but they must do so safely, consistently, and within a design system that preserves brand identity and user trust. A global registry would serve as the shared substrate that connects agent-generated UI descriptions to actual, rendered native components across web, mobile, and desktop surfaces.

Key findings:
- **A2UI (Google)** and **AG-UI (CopilotKit)** are converging on a **declarative, constrained model** where agents emit JSON that references pre-approved components, rather than generating raw HTML/JS.
- **Component registries** (Storybook, Bit.dev, shadcn/ui) are evolving from developer tools to **machine-readable catalogs** with structured metadata, component manifests, and MCP servers for LLM consumption.
- **Open Design System Format (ODSF)** provides a self-contained, agent-consumable bundle for design systems, with tokens, HTML examples, and cross-linked guidelines — all without a build step or SDK.
- **Dynamic UI generation** has bifurcated into **constrained** (A2UI, Vercel AI SDK, Tambo) and **unconstrained** (Claude Artifacts, MCP Apps) approaches. Production systems currently favor constrained for safety.
- **JSON Schema** is the dominant lingua franca for describing UI structure, form fields, and component props, with React JSON Schema Form (RJSF) and ui-schema as mature reference implementations.
- **Vector embeddings** enable semantic component search, allowing agents to find the right component from natural language descriptions rather than exact keyword matching.

## 2. Topic 1: A2UI & AG-UI (Agent-to-User Interface) Concepts

### 2.1 Definitions & Core Concepts

**A2UI (Agent to UI)** is an open-source declarative UI protocol developed by Google (Apache 2.0) for agent-driven interfaces. It allows AI agents to generate rich, interactive UIs that render natively across platforms without executing arbitrary code.

**AG-UI (Agent-User Interaction Protocol)** is a complementary open protocol developed by CopilotKit that standardizes how agent activity and state stream to front-end applications over SSE (Server-Sent Events). It is adopted by Google, LangChain, AWS, Microsoft, Mastra, and PydanticAI.

**How they differ from traditional UI:**

| Dimension | Traditional UI | A2UI / AG-UI |
|-----------|---------------|--------------|
| Generation | Hand-coded by developers | Generated by agents at runtime |
| Transport | N/A (same process) | JSON messages over A2A, AG-UI, SSE, WebSockets |
| Trust model | Trusted (same codebase) | Cross-trust-boundary (remote agents) |
| Rendering | Direct DOM manipulation | Client renders from declarative spec |
| Security | Code review | Pre-approved component catalog + declarative data |
| Portability | Framework-specific | Framework-agnostic (same JSON → React, Flutter, Angular, Lit) |

### 2.2 Key Primitives & Patterns

**A2UI Core Primitives (v0.9.1 / v1.0 candidate):**

| Primitive | Purpose | Example |
|-----------|---------|---------|
| **Surface** | Canvas for components (dialog, sidebar, main view) | `createSurface` with `surfaceId` |
| **Component** | UI element reference from a catalog | `Button`, `Text`, `Card`, `DateTimeInput` |
| **Data Model** | Application state components bind to | `updateDataModel` with `path` and `value` |
| **Catalog** | Available component types and their schemas | JSON file listing allowed components |
| **Message** | JSON object describing a UI action | `surfaceUpdate`, `dataModelUpdate`, `createSurface` |
| **Action** | User interaction sent back to agent | `action` with `event.name` and `context` |
| **Surface Properties** | Theming/styling hints for the client | `primaryColor`, `font`, `styles` (renamed from `theme` in v1.0) |

**A2UI Message Flow:**
1. User sends a message to an AI agent.
2. Agent generates A2UI messages (structure + data).
3. Messages stream to the client application.
4. Client renders using native components (Angular, Flutter, React, etc.).
5. User interacts with the UI, sending actions back to the agent.
6. Agent responds with updated A2UI messages.

**AG-UI Protocol Patterns:**
- **Event-based streaming**: Agent emits events; front-end subscribes.
- **Shared state**: Bi-directional state synchronization between agent and UI.
- **Tool-calling integration**: Agent can invoke tools that return A2UI payloads.
- **Full A2A handshake**: AG-UI implements a full handshake with the A2A protocol for seamless integration.

### 2.3 Key Tools & Frameworks

| Tool / Framework | Role | Description |
|------------------|------|-------------|
| **A2UI** | Protocol (Google) | Declarative JSON spec for agent-generated UIs. v0.9.1 current, v1.0 candidate. |
| **AG-UI** | Protocol (CopilotKit) | Agent-to-frontend event streaming protocol. Day-0 compatible with A2UI. |
| **A2UI Composer** | Visual Editor | Generate A2UI JSON from a visual editor at a2ui-editor.ag-ui.com |
| **A2UI Theater** | Playground | Step through pre-built A2UI streaming scenarios across Lit, React, Angular. |
| **CopilotKit** | Framework | Full-stack framework for in-app agents and generative UI across React, Angular, mobile. |
| **Vercel AI SDK** | SDK | TypeScript toolkit with `streamUI()` for server-streamed components. |
| **Tambo** | SDK | Generative UI SDK for React, purpose-built for streaming AI-generated components. |
| **assistant-ui** | Primitives | TypeScript/React primitives for AI chat with generative-UI support. |
| **Crayon (Thesys)** | Toolkit | MIT-licensed React toolkit built on Radix and shadcn/ui patterns. |
| **mdocUI** | Framework | Streaming generative UI using Markdoc tag syntax. Framework-agnostic core. |
| **Hashbrown** | Framework | Generative UI framework for Angular and React. |
| **Cuttlekit** | Framework | Fully generative UI framework, framework-agnostic, optimized for real-time. |
| **mcp-ui** | Community SDK | TypeScript/Ruby/Python SDK for serving interactive UI over MCP. |

### 2.4 Design Principles

1. **Security First**: Declarative data format, not executable code. Agents can only request components from the client's trusted catalog — no UI injection attacks.
2. **LLM-Friendly**: Flat component list with ID references. Easy to generate incrementally, correct mistakes, and stream.
3. **Framework-Agnostic**: Agent sends abstract component tree; client maps to native widgets.
4. **Separation of Concerns**: Three layers — UI structure, application state, client rendering. Enables data binding, reactive updates, clean architecture.
5. **Progressive Rendering**: Components can be streamed and rendered as they arrive, improving perceived performance.

### 2.5 What A2UI Is NOT

- Not a framework (it is a protocol).
- Not a replacement for HTML (for agent-generated UIs, not static sites).
- Not a robust styling system (client controls styling with limited server-side styling support).
- Not limited to web (works on mobile and desktop).

## 3. Topic 2: Component Registries & Design Systems

### 3.1 How Modern Design Systems Manage Component Registries

Modern design systems have evolved from static documentation sites to **dynamic, machine-readable catalogs** that serve both human developers and AI agents. The trend is toward **component manifests** — structured metadata that describes every component, its props, types, examples, and relationships.

### 3.2 Key Platforms & Approaches

#### Storybook + Chromatic
- **Storybook** is the most widely adopted tool for building and documenting design systems.
- **Component Manifests** (RFC #1, Sept 2025): Storybook is proposing to generate a JSON file (`components-manifest.json`) containing metadata about all components: names, props, types, defaults, descriptions, example code snippets, related components, and MDX documentation pages.
- **MCP Server**: A proposed "Design System MCP Server" would consume these manifests and expose them to LLMs in a token-efficient, human-readable format.
- **Tools**: List all components, get details for components, search components by keywords.
- **Chromatic** provides visual regression testing and PR preview environments for Storybook.

#### Bit.dev
- Bit.dev is a component hosting platform that treats components as first-class packages.
- Each component is independently versioned, documented, and tested.
- Strong dependency graph and component composition model.
- **Relevance to A2UI**: Bit.dev's component isolation and metadata model aligns well with the need for a machine-readable component catalog.

#### Figma Design Systems
- Figma is the dominant design tool for creating and maintaining design systems.
- **Figma Make** (July 2025): Prompt-to-UI inside Figma that builds working interfaces using existing components and design systems.
- **Figma-Context-MCP**: MCP server that provides Figma layout information to AI coding agents.
- **Design tokens** are increasingly exported from Figma to code (e.g., via Style Dictionary, Tokens Studio).
- **Gap**: Figma is visual-first; bridging Figma's visual model to A2UI's declarative JSON model is an open integration challenge.

#### shadcn/ui Registry
- shadcn/ui has a sophisticated registry resolution and fetching system.
- **Registry Index Selection**: Separate indices for styles (e.g., `new-york-v4`) and bases (e.g., `radix`).
- **LRU Caching**: In-memory cache for parsed and validated registry items to avoid repeated disk I/O.
- **Code Transformation Pipeline**: Replaces `export default` with named exports, fixes internal alias imports, and normalizes paths.
- **Entry structure**: File descriptors, dependencies, metadata (CSS variables, Tailwind config, installation instructions).
- **Impact**: shadcn/ui's registry pattern is the de facto standard for AI-generated components; many tools (v0, Magic Patterns) target it as the output format.

### 3.3 Architecture Patterns for Component Registries

| Pattern | Description | Example |
|---------|-------------|---------|
| **Manifest-based** | JSON metadata file listing all components and their properties | Storybook `components-manifest.json` |
| **Index-based** | Registry index files per style/base that map names to entries | shadcn/ui `_registry.ts` |
| **MCP Server** | Model Context Protocol server that exposes registry to LLMs | Storybook MCP, Figma-Context-MCP |
| **Schema Registry** | Centralized schema storage and validation (Cloudera, Confluent) | Schema Registry web server + pluggable storage |
| **CDN-based** | Static JSON files served over HTTP for consumption by agents | shadcn/ui registry JSON |
| **Package-based** | Each component is a versioned npm package with metadata | Bit.dev |

### 3.4 Best Practices for Registry Metadata

A machine-readable component registry should include:
- **Component name** and **slug** (unique identifier)
- **Short and long descriptions** (human and LLM-readable)
- **Props schema** with types, defaults, descriptions, and validation rules
- **Example code snippets** (derived from stories or tests)
- **Related components** (composition graph)
- **Documentation pages** (MDX or markdown)
- **Design tokens** used (color, typography, spacing)
- **Accessibility** notes and ARIA roles
- **Status** (stable, experimental, deprecated)
- **Platform applicability** (web, iOS, Android, desktop)
- **Dependencies** (internal and external packages)

### 3.5 Open Component Registry Standards

There is no single dominant open standard for component registries, but several are emerging:
- **Storybook Component Manifests** (proposed)
- **Open Design System Format (ODSF)** — see Topic 3
- **shadcn/ui Registry Format** (de facto standard for AI tooling)
- **W3C Design Tokens Community Group (DTCG)** — for token interoperability
- **JSON Schema** — for component prop validation (see Topic 5)

## 4. Topic 3: Open Design System Format (ODSF)

### 4.1 What Is Open Design?

Open Design is a movement toward **packaging design systems as self-contained, agent-consumable bundles** that require no SDK, no platform, and no lock-in. The flagship specification is the **Open Design System Format (ODSF) v0.1**, a profile of the Open Knowledge Format (OKF).

The goal: hand a bundle to an agent and say "build this screen, adhere to this design system," and have it produce work that looks and behaves correctly because the system is right there in a form the agent can consume.

### 4.2 Core Concepts

**Bundle Structure:**

```
bundle/
  index.md                      ← bundle-root listing; declares odsf_version
  log.md                        ← chronological change history
  foundations/                  ← the design language: tokens and their meaning
    index.md
    color.md                    ← type: Color
    typography.md               ← type: Typography
    spacing.md                  ← type: Spacing
    elevation.md                ← type: Elevation
    shape.md                    ← type: Shape
    motion.md                   ← type: Motion
  components/                   ← reusable UI elements
    index.md
    button.md                   ← type: Component
    button.example.html         ← self-rendering canonical example (asset)
    card.md
    card.example.html
  patterns/                     ← compositions of components
    index.md
    form.md                     ← type: Pattern
    form.example.html
  behaviors/                    ← interaction and state rules
    index.md
    focus-visible.md            ← type: Behavior
  guidelines/                   ← do/don't principles
    index.md
    color-not-alone.md          ← type: Guideline
    color-not-alone.dont.html   ← counter-example (asset)
  styles/                       ← token set as consumable CSS
    tokens.css                  ← design tokens as CSS custom properties
    components.css              ← styles the example HTML links to
  references/                   ← external sources mirrored as concepts
    index.md
    design-md.md                ← type: Reference
```

### 4.3 Token System

ODSF carries tokens in **two projections of one source of truth**:

1. **Frontmatter `tokens`**: Human- and agent-readable YAML definitions on the foundation concept.
2. **`styles/tokens.css`**: The same tokens as CSS custom properties, ready to drop into code.

**Token shape example:**
```yaml
tokens:
  colors:
    primary: "#3b5bdb"
    on-primary: "#ffffff"
    surface: "#ffffff"
    text: "#1a1b1e"
  spacing:
    sm: "8px"
    md: "16px"
    lg: "24px"
  typography:
    body:
      fontFamily: "Inter, system-ui, sans-serif"
      fontSize: "16px"
      lineHeight: "1.5"
```

**CSS projection:**
```css
:root {
  --colors-primary: #3b5bdb;
  --colors-on-primary: #ffffff;
  --spacing-md: 16px;
  --radius-md: 8px;
}
```

**Token references**: Components reference foundation tokens with `{group.name}` syntax:
```yaml
# components/button.md
tokens:
  button-primary:
    backgroundColor: "{colors.primary}"
    textColor: "{colors.on-primary}"
    radius: "{radius.md}"
    padding: "{spacing.sm} {spacing.md}"
```

### 4.4 Component Models

**Concept Types** (ODSF vocabulary):

| Type | Description |
|------|-------------|
| **Color** | Palette, semantic roles, theming |
| **Typography** | Type families, scale, text roles |
| **Spacing** | Sizing scale |
| **Elevation** | Shadow/depth system |
| **Shape** | Corner radii, border treatment |
| **Motion** | Durations, easings, animation principles |
| **Layout** | Grid, breakpoints, responsive rules |
| **Component** | Reusable UI element (button, input, card) |
| **Pattern** | Composition of components (form, empty state, page shell) |
| **Behavior** | Interaction/state rule (focus handling, loading) |
| **Guideline** | Do/don't principle with rationale |
| **Accessibility** | Contrast, focus order, ARIA, motion-reduction |
| **Voice** | Tone and microcopy guidance |
| **Design System** | Bundle-level overview |
| **Reference** | External source mirrored into the bundle |

### 4.5 Companion Assets

Every concept SHOULD ship concrete, self-rendering artifacts:
- **`<concept>.example.html`** — canonical, correct usage; self-rendering in a browser with no build step.
- **`<concept>.do.html` / `<concept>.dont.html`** — correct/counter-example pairs for guidelines.
- **`<concept>.css`** — styles specific to the concept.

### 4.6 Consuming an ODSF Bundle

How an agent should use a bundle:
1. **Orient** — Read `index.md` and the `Design System` overview.
2. **Pull foundations** — Load tokens (or link `styles/tokens.css`).
3. **Descend by need** — Follow `index.md` listings and cross-links to relevant components, patterns, and behaviors.
4. **Copy from assets** — Reproduce structure, class names, and attributes from `*.example.html`.
5. **Degrade gracefully** — If a pattern is missing, compose it from components; if a component is missing, use the nearest typed sibling or a primitive.
6. **Respect rules** — Honor guidelines, accessibility concepts, and `*.dont.html` counter-examples.

### 4.7 Relationship to Other Formats

- **OKF** gives the container (bundle, concept, links, indexes, logs, versioning).
- **design.md** gives the seed of the content (frontmatter tokens, reference syntax, variant/state entries).
- **ODSF** adds the rest: pattern/behavior/guideline/accessibility types, runnable `tokens.css`, companion HTML/CSS assets, and a graph an agent can navigate from task to rule.
- **DTCG / Tailwind** — ODSF is compatible with W3C DTCG format and Tailwind but does not redefine them.

### 4.8 Real-World ODSF Bundles

Open Design (open-design.ai) provides pre-built design system bundles for major brands, all conforming to ODSF:
- **OpenAI** — Calm, near-monochrome system anchored in deep teal-black with generous white space.
- **GitHub** — Code-forward platform, blue-on-white precision, Primer foundations.
- **Vercel** — Black and white precision, Geist font.
- **Cursor** — AI-first code editor, sleek dark interface, gradient accents.
- **Mission Control** — Space/aerospace mission monitoring, dark command center, amber telemetry.
- **Glassmorphism** — Frosted glass effect with translucent layers.
- **Coinbase Design System (CDS)** — 100+ production components, open-sourced Oct 2025.

## 5. Topic 4: Agentic Application Metadata & Dynamic UI Generation

### 5.1 The Landscape of Dynamic UI Generation

Dynamic UI generation has bifurcated into two approaches:

| Approach | Description | Examples | Pros | Cons |
|----------|-------------|----------|------|------|
| **Constrained** | AI selects from registered components | A2UI, Vercel AI SDK `streamUI()`, Crayon, assistant-ui, Tambo, v0.dev | Safe, consistent, predictable | Less flexible, requires component development upfront |
| **Unconstrained** | AI generates raw HTML/CSS/JS directly | Claude Artifacts, MCP Apps, OpenAI Canvas, Google GenUI research | Powerful, flexible, emergent | Needs sandboxing (iframe, E2B), potential inconsistency, security risks |

Google's research ("Generative UI: LLMs are Effective UI Generators," 2025) found users preferred AI-generated HTML/CSS over markdown 83% of the time, calling UI generation an "emergent capability" requiring no UI-specific training. However, the 2025–2026 production wave (AG-UI, A2UI, MCP Apps, Crayon, assistant-ui) has leaned toward **constrained / declarative** output for safety and consistency.

### 5.2 Key Tools & Platforms

#### Component Generators ("Describe a component, get code")
- **v0.dev** (Vercel) — Generate React/Tailwind components from prompts; shadcn/ui integration.
- **OpenUI** (Weights & Biases) — Describe UI in natural language and see it rendered live.
- **openv0** — Open source v0 clone.
- **Figma Make** — Prompt-to-UI inside Figma using existing components and design system.
- **Google Stitch** — Generates web and mobile UI designs plus front-end code from prompts, screenshots, or sketches.
- **Magic Patterns** — Generate and iterate on React UI from prompts; can learn and apply existing design systems.
- **Subframe** — AI design tool built for code: compose production-ready React/Tailwind components on a canvas.
- **Screenshot-to-Code** — Convert screenshots or designs to HTML/React/Vue code.
- **tldraw make-real** — Turn a wireframe into a working React component.

#### App Builders ("Describe an app, get a deployable project")
- **Bolt.new** (StackBlitz) — Generate full-stack applications in-browser with live preview.
- **Lovable** — Generate and deploy web apps from descriptions.
- **Cofounder** — Open source full-stack app generator with generative UI, databases, and APIs.

#### AI Products with Built-In UI Generation
- **Claude** (Anthropic) — Artifacts generates interactive React/HTML UIs in chat. July 2025 update turned artifacts into lightweight apps that can call Claude through an API and be shared.
- **ChatGPT** (OpenAI) — Canvas supports UI generation and editing.
- **Amazon Rufus** — Conversational shopping assistant that generates intent-based interfaces (product cards, filter options) through conversation.

#### MCP Tools for UI
- **Figma-Context-MCP** — Provides Figma layout information to AI coding agents.
- **21st.dev Magic MCP** — Generate UI components from prompts with design-system awareness.
- **shadcn-ui-mcp-server** — Helps LLMs understand shadcn/ui component structure.
- **MUI MCP** — MCP server for MUI docs and code examples.
- **Storybook MCP** — Exposes component information and workflows from local Storybook.
- **Playwright MCP** — Browser automation and UI regression testing workflows.

### 5.3 Frameworks & SDKs for Generative UI

| Framework | Purpose | Key Features |
|-----------|---------|--------------|
| **Vercel AI SDK** | TypeScript toolkit | `streamUI()` for server-streamed components; multi-provider; v5 adds React/Vue/Svelte parity; SSE-based streaming. |
| **Tambo** | Generative UI SDK for React | Purpose-built for streaming AI-generated components. |
| **Hashbrown** | Framework | Generative UI in Angular and React. |
| **Cuttlekit** | Framework | Fully generative UI, framework-agnostic, optimized for real-time. |
| **mdocUI** | Framework | Streaming generative UI using Markdoc `{% %}` tags; 24 theme-neutral components; Zod schema validation. |
| **CopilotKit** | Full-stack framework | In-app agents and generative UI across React, Angular, mobile, Slack; makers of AG-UI. |
| **assistant-ui** | Primitives | TypeScript/React for AI chat; first-class generative-UI primitive from allow-listed components. |
| **Thesys C1** | OpenAI-compatible API | Returns rendered UI instead of text; paired with Crayon React toolkit. |
| **OpenUI Lang** | DSL + Runtime | MIT-licensed, streaming-first DSL and React runtime; compact, token-efficient alternative to component JSON. |
| **Instructor** | Structured output | Extraction library useful with UI schemas for reliable generation. |
| **Mitosis** | Builder.io | Write components once, compile to React/Vue/Svelte/etc. |

### 5.4 Security & Sandboxing for Generated UI

When UIs are model-generated, outputs must be treated as **untrusted**:
- **OWASP Top 10 for LLM Applications** — Threat model for prompt injection, insecure tool use, data leakage.
- **Content Security Policy (CSP)** — Core browser control for limiting script execution.
- **iframe sandbox** — Sandboxed iframes for containing untrusted HTML/JS previews.
- **Trusted Types** — Mitigation against DOM XSS when inserting generated HTML.
- **DOMPurify** — Widely used HTML sanitizer for untrusted/generated markup.
- **Sandpack** (CodeSandbox) — In-browser code sandboxing patterns.
- **E2B** — Sandboxed code execution environments for running untrusted generated code.

### 5.5 How the Layers Fit Together

These specs are complementary, standardizing different layers of the same stack:

| Layer | What it Standardizes | Examples |
|-------|---------------------|----------|
| Transport / events | How agent activity and state stream to the UI | AG-UI |
| UI description format | How the UI itself is described in the payload | A2UI (declarative components), MCP Apps (HTML resources) |
| In-client rendering surface | Where generated UI renders in a host chat app | MCP Apps, OpenAI Apps SDK, mcp-ui |

## 6. Topic 5: JSON Schema for UI Generation

### 6.1 Core Concepts

JSON Schema is a standard vocabulary for annotating and validating JSON documents. In the UI generation context, it serves as the **contract** between the agent (which produces UI descriptions) and the client (which renders them).

JSON Schema is used for:
- **Form generation**: Automatically generating forms from data schemas.
- **Component prop validation**: Defining what props a component accepts, their types, defaults, and constraints.
- **Structured output**: Constraining LLM outputs to valid, parseable JSON.
- **Catalog definition**: Describing the component registry (what components exist, what they do, what they need).

### 6.2 Key Tools & Libraries

| Tool / Library | Description | Key Features |
|----------------|-------------|--------------|
| **React JSON Schema Form (RJSF)** | Most popular React library for schema-driven forms | Extensive customization; UI schema for presentation; conditional fields; custom widgets; validation in schema. |
| **ui-schema/ui-schema** | Headless React components for schema-driven UIs | Add any design system or custom widget; auto-rendering; flexible translation; modular core; isomorphic code. |
| **JSON Schema Builder** (M3MONs) | Visual tool for creating JSON Schema and UI Schema | Click-to-add interface; real-time preview; schema export; field validation; grid layout. |
| **Schematik Forms** (Deloitte) | Schema-based form builder for React | Error summaries; custom renders; show/hide rules; BYO components. |
| **DronaHQ JSON Schema Form** | Low-code form builder | Programmatic schema modifications; predefined default data; input validation; customizable UI elements. |
| **form-render** (Alibaba) | React-based form solution | Uses Ant Design as visual theme; low-code friendly. |
| **formily** (Alibaba) | Cross-end form solution | Logic cross-framework; @formily/core for state management; @formily/react for UI bridge. |
| **vue-json-schema-form** | Vue.js form renderer | Supports Vue 3; JSON Schema driven. |
| **Zod** | TypeScript schema validation | Runtime type checking; popular for UI output shapes in generative UI frameworks. |
| **AJV** | JSON Schema validator | Fast validation; bridges errors to react-hook-form. |
| **Outlines** | Structured generation | JSON/grammar constraints for model outputs. |
| **Guidance** | Grammar-oriented prompting | Constraint-oriented prompting for structured outputs. |

### 6.3 Architecture Patterns

#### Schema-Driven Form Generation
```
JSON Schema → Form Renderer → UI Components
     ↓              ↓              ↓
Data structure   Widget mapping   Input fields, validation, layout
```

**Example flow:**
1. Define data schema (e.g., user profile with `name`, `email`, `age`).
2. Define UI schema (e.g., `email` field uses `email` widget, `age` uses `number` with slider).
3. Form renderer maps schema types to UI widgets.
4. User input is validated against the schema.
5. Submitted data conforms to the schema.

#### Schema-Driven Component Catalog
```
Component Catalog Schema → Registry → Agent Selection
     ↓                         ↓              ↓
Component definitions    Machine-readable   Agent picks component
(names, props, types)    index/manifest     by name from catalog
```

### 6.4 Best Practices

- **Separate data schema from UI schema**: Data schema describes the shape; UI schema describes presentation.
- **Use structured output for agent generation**: Force the LLM to emit valid JSON that matches a schema (e.g., using OpenAI function calling, Zod, or Outlines).
- **Validate at runtime**: Use AJV or Zod to validate agent-generated UI descriptions before rendering.
- **Version schemas**: Component schemas may evolve; version them alongside the registry.
- **Support conditional logic**: Use `oneOf`, `anyOf`, `if/then/else` in JSON Schema for conditional field visibility.
- **Include examples**: Embed example data in the schema (`examples` keyword) to help agents understand expected values.

### 6.5 Integration with A2UI

A2UI's catalog concept aligns closely with JSON Schema:
- The **catalog** is a JSON file listing allowed components.
- Each component entry can include a **JSON Schema** describing its props, types, and constraints.
- The agent generates **A2UI messages** that reference components by name and provide props that conform to the schema.
- The client validates incoming A2UI messages against the catalog schema before rendering.

## 7. Topic 6: Embeddings & Semantic Component Search

### 7.1 Core Concepts

**Semantic component search** uses vector embeddings to find UI components based on natural language descriptions rather than exact keyword matches. This is critical for A2UI because:
- Agents describe UI needs in natural language (e.g., "a card for displaying a restaurant with image, name, rating, and a book button").
- The registry must map this description to the most appropriate component(s).
- Traditional keyword search fails when descriptions use synonyms, metaphors, or domain-specific language.

### 7.2 How Semantic Search Works

1. **Text processing and embedding models** convert unstructured text (component descriptions, docstrings, prop names, usage examples) into dense numerical vectors.
2. **Vector storage and indexing** stores these embeddings in a database optimized for similarity search (FAISS, Annoy, Qdrant, Elasticsearch with vector plugins, OpenSearch).
3. **Query handling and ranking** converts the agent's natural language query into a vector, performs approximate nearest neighbor (ANN) search, and returns the most similar components.

### 7.3 Key Technologies

| Technology | Role | Description |
|------------|------|-------------|
| **Sentence-BERT** | Embedding model | Generates sentence embeddings for semantic similarity. |
| **Universal Sentence Encoder** | Embedding model | Google model for encoding sentences into vectors. |
| **OpenAI text-embedding-3-large** | Embedding model | High-quality embeddings for semantic search (used in code Q&A systems). |
| **FAISS** (Facebook) | Vector index | Efficient similarity search and clustering of dense vectors. |
| **Annoy** (Spotify) | Vector index | Approximate nearest neighbors using tree structures. |
| **Qdrant** | Vector database | Open-source vector database with filtering and payload storage. |
| **Elasticsearch** (with vector plugins) | Search engine | Combines keyword and vector search in a single query. |
| **OpenSearch** | Search engine | AWS-managed vector search with k-NN plugin. |
| **Milvus** | Vector database | Distributed vector database for AI applications. |
| **Hugging Face Transformers** | Model hub | Provides pre-trained models for embedding generation. |
| **nomic-embed-text-v2-moe** | Local embedding model | Can run entirely on local hardware for privacy. |
| **all-MiniLM-L6-v2** | Local embedding model | Lightweight, fast, suitable for local deployment. |

### 7.5 Advanced Techniques

- **HyDE (Hypothetical Document Embeddings)**: Generate a hypothetical ideal component description from the query, embed that, and search against the index. Improves recall for vague queries.
- **MMR (Maximal Marginal Relevance)**: Balance relevance with diversity in search results, so the agent sees a variety of component options rather than near-duplicates.
- **Hybrid search**: Combine vector similarity with keyword matching (e.g., BM25) for better precision.
- **Multi-modal embeddings**: If components include screenshots or previews, use vision-language models (e.g., CLIP) to embed both text and image.
- **Re-ranking**: Use a cross-encoder model to re-rank the top-k results from the vector search for higher precision.

### 7.6 Real-World Example: Semantic Search for React Components

A 2025 Hackernoon article described building an AI-powered UI search system using:
- Vector embeddings for component descriptions and code.
- HyDE for query expansion.
- MMR for result diversity.
- Integration with React component libraries for improved discovery.

### 7.7 Integration with A2UI Registry

For a global A2UI component registry, semantic search would work as follows:
1. Every registered component is indexed with:
   - Name, description, tags
   - Prop schemas (as text)
   - Usage examples (code snippets)
   - Accessibility notes
   - Related components
2. An agent submits a natural language query: "I need a form for booking a restaurant table with date, time, and number of guests."
3. The registry embeds the query and performs ANN search.
4. Returns ranked component suggestions: `Form`, `DateTimeInput`, `NumberInput`, `Button`, `Card`.
5. The agent selects components and generates an A2UI message using their catalog IDs.

### 8.1 The Vision

A **Global A2UI Component Registry** is a federated, machine-readable catalog of UI components that any agent can discover, query, and compose into interfaces — regardless of the target platform (web, mobile, desktop) or framework (React, Flutter, Angular, SwiftUI).

### 8.3 Design Decisions for the Registry

| Decision | Rationale | Recommendation |
|----------|-----------|----------------|
| **Constrained vs Unconstrained** | Safety and consistency are paramount for a global registry serving multiple organizations and trust boundaries. | **Constrained** — agents select from pre-registered components. |
| **Catalog Format** | Must be LLM-friendly, machine-readable, and validateable. | **JSON Schema** for component definitions + **ODSF** for design system bundles. |
| **Transport** | Must support streaming, bidirectional communication, and cross-platform compatibility. | **AG-UI** for event streaming + **A2UI** for UI payload format. |
| **Search** | Agents won't know exact component names; they describe needs in natural language. | **Semantic search** with vector embeddings + hybrid keyword fallback. |
| **Styling** | Each client has its own brand; the registry should not impose styles. | **Design tokens** (ODSF) + **client-side theme mapping** (surfaceProperties in A2UI). |
| **Versioning** | Components evolve; breaking changes must be managed. | Semantic versioning per component + migration guides in catalog. |
| **Security** | Remote agents are untrusted; code execution is unacceptable. | **Declarative only** (no raw HTML/JS), **pre-approved catalog**, **CSP** + **sandboxing** for previews. |
| **Federation** | No single organization should own the registry. | **Federated model** — multiple registries expose a common API; components can be cross-referenced. |

### 8.4 The Role of ODSF in the Registry

ODSF bundles can serve as the **design system layer** of the global registry:
- Each design system (e.g., "Material Design 3", "Carbon", "Primer", "OpenAI") is packaged as an ODSF bundle.
- The registry indexes the bundle's components, tokens, patterns, and guidelines.
- Agents can discover not just components but also **how to use them correctly** (do/don't pairs, accessibility rules, behavior constraints).
- The self-rendering HTML examples in ODSF provide a canonical reference that both agents and renderer implementers can test against.

### 8.5 The Role of JSON Schema in the Registry

JSON Schema serves as the **contract layer**:
- Every component in the registry has a JSON Schema describing its props, types, defaults, and constraints.
- The agent's A2UI output is validated against this schema before rendering.
- Form components can be auto-generated from the schema for rapid testing.
- The registry's API itself is described with OpenAPI (which uses JSON Schema), making it discoverable and type-safe.

### 8.6 The Role of Semantic Search in the Registry

Semantic search is the **discovery layer**:
- An agent describes a UI need in natural language (e.g., "a dashboard widget showing KPIs with a trend indicator").
- The registry embeds this query and searches across component descriptions, prop names, usage examples, and pattern documentation.
- Returns ranked components: `KPICard`, `TrendIndicator`, `StatCard`, `MetricPanel`.
- The agent can refine: "show me ones with sparkline charts" → narrows to `KPICardWithSparkline`.
- Hybrid search (vector + keyword) ensures exact matches for known component names are prioritized.

### 8.7 Integration with Existing Ecosystems

The global registry should integrate with, not replace, existing tools:
- **Storybook**: Consume `components-manifest.json` as a registry input; Storybook MCP server exposes components to agents.
- **shadcn/ui**: Support shadcn's registry format as a first-class input; allow agents to generate shadcn-compatible components.
- **Figma**: Figma-Context-MCP bridges visual design to the registry; ODSF tokens can be synced from Figma.
- **Bit.dev**: Federate with Bit.dev's component packages; import metadata and examples.
- **npm / JSR**: Components can be published as packages with registry metadata in their package.json.

## 9. Recommendations

### 9.1 For Registry Implementers

1. **Adopt A2UI as the UI description format** and AG-UI as the transport protocol. These are the most mature, widely adopted open standards in this space.
2. **Use JSON Schema for component definitions** and validate all agent-generated UI payloads at runtime.
3. **Package design systems as ODSF bundles** to make them self-contained, portable, and agent-consumable.
4. **Implement semantic search** with vector embeddings (e.g., using Qdrant or FAISS) for natural language component discovery.
5. **Support hybrid search** (vector + keyword) for the best of both worlds.
6. **Federate the registry** — no single organization should own it. Allow multiple registries to interoperate via a common API.
7. **Provide self-rendering examples** (HTML/CSS) for every component so agents and renderer implementers have a canonical reference.
8. **Include accessibility and guideline metadata** as first-class properties (not afterthoughts) to ensure generated UIs are inclusive.
9. **Version everything** — components, schemas, tokens, and design systems. Support semantic versioning and deprecation paths.
10. **Security by default** — no raw code execution, pre-approved catalogs only, sandbox previews, CSP headers, and DOMPurify for any HTML rendering.

### 9.2 For Agent Developers

1. **Use constrained generation** — select from the registry rather than generating raw HTML/JS. This ensures consistency and safety.
2. **Leverage semantic search** — don't hardcode component names; use natural language descriptions to discover the right component.
3. **Respect design system bundles** — when a client specifies a design system, consume its ODSF bundle and follow its tokens, guidelines, and do/don't rules.
4. **Validate output** — use JSON Schema or Zod to validate A2UI messages before sending them to the client.
5. **Support progressive rendering** — stream A2UI messages so the client can render incrementally, improving perceived performance.
6. **Handle degradation gracefully** — if a requested component is not in the catalog, fall back to the nearest available primitive or compose from smaller components.

### 9.3 For Client Application Developers

1. **Maintain a trusted component catalog** — only render components from your pre-approved list. The registry suggests; you decide.
2. **Implement an A2UI renderer** — map A2UI JSON to your native UI framework. Open-source renderers exist for React, Angular, Flutter, and Lit.
3. **Apply your design system** — use ODSF tokens or your own token system to style rendered components so they feel native to your app.
4. **Sanitize everything** — treat agent-generated UI as untrusted. Validate schemas, sanitize any HTML, and use CSP.
5. **Implement action handlers** — when the user interacts with an A2UI component, send the action back to the agent via AG-UI or A2A.

### 9.4 Open Questions & Future Work

1. **How do we handle real-time collaboration** when multiple agents are generating UI for the same surface?
2. **How do we bridge visual design tools** (Figma, Sketch) with declarative A2UI specs in a bidirectional way?
3. **What is the governance model** for a federated global registry? Who approves components? Who moderates?
4. **How do we handle platform-specific capabilities** (e.g., iOS widgets, Android composables) within a cross-platform spec?
5. **Can we use multi-modal embeddings** (text + screenshot) to improve component search and recommendation?
6. **How do we measure and benchmark** the quality of agent-generated UIs? (See Design2Code, WebArena benchmarks.)
7. **What is the business model** for component authors in a global registry? Micropayments? Attribution?
8. **How do we ensure accessibility** at scale when UIs are generated dynamically by agents with varying levels of understanding?
