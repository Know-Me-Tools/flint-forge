# p3-c003 — GraphQL Introspection Merge: pg_graphql ∪ Subscription SDL

## Change ID
`p3-c003-introspection-merge`

## Phase
`p3-graphql-hybrid-engine`

## Priority
P0 — Requires p3-c001 (pg_graphql passthrough) AND p3-c007 (subscription schema)

## Problem Statement

`/__schema` and `__type` introspection currently falls through to pg_graphql
alone. Subscription types generated by `GraphQlCompiler` (p3-c007) are
invisible to clients performing introspection. GraphQL tooling (GraphiQL,
Apollo Studio, typed code generation) will not discover subscription fields.

Per §3.2: "Introspection: merged union of pg_graphql schema ∪ sibling
subscription SDL."

## Scope

### In Scope
- Implement `IntrospectionMerger`: takes the pg_graphql JSON introspection
  response and the async-graphql subscription SDL, merges them into a single
  combined `__Schema` response
- Wire the merger into the `POST /graphql` handler: when the incoming request
  is an introspection query (`__schema` or `__type` field selection), intercept
  and return the merged schema instead of the raw pg_graphql response
- Detect introspection queries reliably (heuristic: query body contains
  `__schema` or `__type` at the root operation level)

### Out of Scope
- Subscription resolver bodies (p3-c002)
- WebSocket handler (p3-c004)
- Any mutation of the pg_graphql SDL itself

## Design

### IntrospectionMerger (fdb-postgres or fdb-app)

```rust
pub struct IntrospectionMerger;

impl IntrospectionMerger {
    /// Merge pg_graphql introspection JSON + async-graphql subscription SDL types.
    ///
    /// Algorithm:
    ///   1. Parse pg_graphql result: extract `data.__schema.types[]` and
    ///      `data.__schema.subscriptionType`
    ///   2. Build subscription types from async_graphql::dynamic::Schema
    ///      using schema.sdl() and parse into a set of type definitions
    ///   3. Append subscription types to pg_graphql `types[]` (dedup by `name`)
    ///   4. Set `data.__schema.subscriptionType` = { "name": "Subscription" }
    ///      if not already present
    ///   5. Return merged JSON
    pub fn merge(
        pg_result: serde_json::Value,
        subscription_schema: &async_graphql::dynamic::Schema,
    ) -> serde_json::Value
}
```

### Introspection detection

```rust
fn is_introspection_query(body: &str) -> bool {
    // Heuristic: fast path for the most common introspection patterns.
    // Covers: `{ __schema { ... } }`, `query { __schema }`, `__type(name: "X")`
    body.contains("__schema") || body.contains("__type")
}
```

This is a best-effort heuristic. False positives (regular queries containing
`__schema` as a substring of a field name) are acceptable — they cost one
merge call, not a data-correctness problem.

### Integration point: POST /graphql handler

```rust
// In fdb-gateway/src/main.rs handle_graphql_query():
if is_introspection_query(&body) {
    let compiled = state.state_manager.current();
    if let Some(sub_schema) = &compiled.subscription_schema {
        // 1. Execute pg_graphql introspection
        let pg_result = executor.execute(req, &rls).await?;
        // 2. Merge with subscription SDL
        let merged = IntrospectionMerger::merge(pg_result, sub_schema);
        return Ok(Json(merged));
    }
    // If no subscription schema compiled yet, fall through to raw pg_graphql
}
```

### Layer placement

`IntrospectionMerger` is pure JSON manipulation with no IO — place in `fdb-app`
(or a new module `fdb-app::graphql::introspection`). It imports
`async_graphql::dynamic::Schema` for SDL extraction.

This is the ONLY place `fdb-app` touches async-graphql — the dependency is
narrow and justified. `fdb-gateway` orchestrates; `fdb-app` implements the
merge logic.

## Security Contracts
- The introspection merge MUST NOT expose internal implementation types
  (e.g., Rust struct names) beyond what pg_graphql already exposes
- The merged type list MUST be deduped by name — if pg_graphql and the
  subscription schema both define a type with the same name, the pg_graphql
  version wins (it is more authoritative about the DB schema)
- `RlsContext` is still required to call pg_graphql introspection — even
  introspection queries are authenticated and scoped per-tenant

## Acceptance Criteria
- `IntrospectionMerger::merge()` implemented with dedup-by-name logic
- `is_introspection_query()` helper implemented
- `POST /graphql` handler wires introspection detection + merger
- `data.__schema.subscriptionType` present in merged response
- Subscription type names visible in merged `types[]` array
- Unit test `test_introspection_merge_adds_subscription_types`:
  - Build minimal pg_graphql JSON introspection output
  - Build minimal async-graphql subscription schema
  - Call `IntrospectionMerger::merge()`
  - Assert `subscriptionType` present and subscription type names in `types[]`
- `cargo check --workspace` GREEN; clippy pedantic passes
